# AGENTS.md

- Gemini dispatch 必须单线程，绝不并行。
- 原因不是实现偏好，而是同一轮任务可能读写同一份仓库文件；并行会制造竞争、重复回复和不可预测副作用。
- 邮件与飞书都必须复用同一个串行 dispatch 入口；不要绕过 dispatch 直接并发拉起多个 agent。
- 非执行脚本类邮件，发邮件前必须先把完整草稿给用户确认；未获确认，不得发送。
- 默认优先使用“附件驱动”邮件协作：给对方发送 `ps1` 附件而不是在正文堆命令；要求对方执行脚本并把脚本源码与执行结果 `txt` 一起作为附件回传。
- 执行脚本类邮件默认依赖标题关键字触发；正文不需要展开说明，只要几个字即可，不要在正文堆执行指令。
- 执行脚本类邮件不需要逐字确认正文；只需要用户确认标题和是否发送，然后即可发出。
- 执行脚本尽量沿 happy path 多做一点：如果前置条件已满足，就顺手完成下一个低风险、可逆的小步骤，不要每次只停在探测。
- 本机手动重启或重新拉起 `glaw.exe` 时，工作目录固定为 `~/my-claw/`；默认先 `cd ~/my-claw/` 再启动。
- 升级链路里 `kill`、替换 exe、重新启动属于最后一步；在这之前只做准备工作，不要提前触碰现有 `glaw.exe` 进程。
- 远端升级链路的最终切换步骤既定细节：先定位应被替换的 `glaw.exe` 进程 PID；替换目标路径固定为 `~/bin/glaw.exe`；重启时先 `cd ~/g-claw/` 再启动。
- 所有通过邮件执行链路运行的 `.py` 脚本，默认在入口显式设置 `sys.stdout.reconfigure(encoding="utf-8")`，避免中文 stdout 在 Windows 上乱码。
- 执行 Python 脚本时默认直接使用 `python`，不要使用 `uv`；技能目录里的 Python 脚本也遵守这个规则。
- 如果脚本只会执行一次或两次，放在 `./tmp/scripts/`；只有长期复用、稳定沉淀的脚本才放在 `./scripts/`。
- 在真正切换前，补做邮件相关前置检查：读取对端 `~/.env` 与 `~/g-claw/.env`，输出所有 key 名包含 `SENDER` 的配置项，再决定切换。
- 发信之后，默认先等待 30 秒，再检查该对象的最新一封邮件，确认是否已有回信或执行结果返回。
- 对方回传后，优先阅读附件里的脚本源码和结果 `txt`；如果发现问题，下一轮直接继续发修正后的脚本附件，不要重新退回成纯正文命令协作。
- 附件脚本默认把生成的结果文件写到当前工作目录，不要默认写到 `$HOME`；这样回传附件时路径更稳定，也更不容易读错文件。
